# ¿Qué es EC2?

**EC2 = Elastic Compute Cloud**

Es un servicio de **Infraestructura como Servicio (IaaS)** que permite alquilar capacidad de cómputo (servidores virtuales) en la nube de AWS.

- En un centro de datos empresarial tradicional, las aplicaciones se implementan en **servidores físicos**.
- En el Cloud, ese rol lo cumplen las **instancias EC2**, que se alquilan bajo demanda.

Conocer EC2 es fundamental para entender cómo funciona el Cloud de AWS.

---

# Características de las instancias EC2

- **Balanceo de carga y autoescalado** para varias instancias EC2.
- Escala la capacidad computacional **arriba o abajo automáticamente** según la demanda.
- **Automatización del aprovisionamiento**: administra la conexión y configuración de red de las instancias.
- **Variedad de configuraciones** para optimizar rendimiento, memoria, almacenamiento, etc. según el caso de uso.

## Casos de uso

- Procesamiento de datos y análisis.
- Configuración como servidores HTTP (web).
- Aplicaciones de alto rendimiento (high performance).

---

# Datos de usuario (User Data) de EC2

Es posible arrancar una instancia ejecutando un **script de datos de usuario** (User Data).

- **Bootstrapping**: lanzar comandos automáticamente cuando una máquina se inicia.
- El script **solo se ejecuta una vez**, en el primer arranque de la instancia.
- Se ejecuta con permisos de usuario **root**.

## Usos típicos del User Data

- Instalar actualizaciones del sistema.
- Instalación de software.
- Descarga de archivos comunes desde Internet.
- Cualquier tarea de configuración inicial automatizable.

## Ejemplo (Linux)

```bash
#!/bin/bash
yum update -y
yum install -y httpd
systemctl start httpd
systemctl enable httpd
echo "<h1>Hola Mundo desde $(hostname -f)</h1>" > /var/www/html/index.html
```

> Documentación oficial: https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/user-data.html

# Grupos de seguridad (Security Groups)

> 🚫 Es como el **guardia de seguridad en la puerta de un club**: controla quién entra y quién sale.

Un grupo de seguridad funciona como un **firewall virtual** para las instancias EC2.

## Características

- Controlan cómo se permite el tráfico **de entrada y de salida** de las instancias.
- Al lanzar una instancia se le puede asignar **uno o varios** grupos de seguridad.
- Los grupos de seguridad **solo contienen reglas de permiso** (no reglas de denegación explícita).
- Las reglas pueden hacer referencia a:
  - **IP** (rangos de IPv4 o IPv6)
  - **Otro grupo de seguridad**

## Qué regulan exactamente

- El acceso a **puertos** específicos.
- **Rangos de IP autorizados** (IPv4 e IPv6).
- **Tráfico de entrada** (desde fuera hacia la instancia).
- **Tráfico de salida** (desde la instancia hacia fuera).

### Ejemplo de reglas

**Grupo de seguridad 1** (permite SSH desde una IP concreta):

| Fuente | Protocolo | Puerto |
|---|---|---|
| 198.51.100.0/24 | TCP | 22 |
| Subred A (CIDR) | Todos | Todos |

**Grupo de seguridad 2** (permite tráfico desde otra subred):

| Fuente | Protocolo | Puerto |
|---|---|---|
| Subred B (CIDR) | Todos | Todos |
| Subred A (CIDR) | TCP | 22 |

Ambos grupos, por defecto, permiten salida total:

| Destino | Protocolo | Puerto |
|---|---|---|
| 0.0.0.0/0 | Todos | Todos |

## Cosas importantes a saber

- Un grupo de seguridad puede adjuntarse a **múltiples instancias**.
- Buena práctica: mantener un **grupo de seguridad separado** solo para el acceso SSH.
- Si la aplicación **no es accesible** (error de tiempo de espera / timeout) → suele ser un **problema del grupo de seguridad**.
- Si la aplicación da un error de **"conexión rechazada"** → suele ser un **error de la propia aplicación** (no se ha lanzado correctamente).

### Configuración por defecto

- ❌ Todo el **tráfico de entrada** está bloqueado.
- ✅ Todo el **tráfico de salida** está autorizado.

# EC2 Instance Connect

Permite **conectarse a una instancia EC2 desde el navegador**, sin necesidad de configurar credenciales de seguridad localmente.

- AWS carga una **clave temporal** en la instancia EC2.
- Es necesario que el **puerto 22** siga abierto en el grupo de seguridad de la instancia.
- Disponible para distintos tipos de instancia; si no está instalado, se puede configurar siguiendo la guía oficial: https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/ec2-instance-connect-set-up.html

# SSH (Secure Shell)

Protocolo de red que permite la **comunicación segura** entre dos sistemas, normalmente para acceder a una terminal remota.

- Proporciona **cifrado y autenticación robusta** para proteger los datos transmitidos.
- Fundamental para la **administración remota** de servidores y dispositivos de red.

## Cómo funciona

1. **Inicio de conexión**: el cliente inicia la conexión y el servidor la acepta (puerto **22** por defecto).
   ```
   ssh joan@192.168.1.10
   ```
2. **Cifrado**: se utilizan técnicas de cifrado (algoritmos como **AES** y **RSA**) para proteger los datos, evitando que sean interceptados por terceros. Se produce un intercambio de claves públicas.
3. **Autenticación**: el usuario se autentica mediante **contraseña** o **claves SSH** (par de claves pública/privada).

## Conexión a EC2 mediante SSH según sistema operativo

| Sistema Operativo | SSH nativo | Putty | EC2 Instance Connect |
|---|:---:|:---:|:---:|
| macOS | ✅ | - | ✅ |
| Linux | ✅ | - | ✅ |
| Windows < 10 | ❌ | ✅ | ✅ |
| Windows ≥ 10 | ✅ | ✅ | ✅ |

### macOS / Linux (y Windows ≥ 10)

Se puede configurar el archivo `~/.ssh/config` de OpenSSH para facilitar las conexiones SSH a instancias EC2.

```bash
ssh -i /path/key-pair-name.pem instance-user-name@instance-public-dns-name
```

Conexión: **Usuario → SSH (puerto 22) → Instancia EC2 Linux (IP pública)**

### Windows < 10

Requiere configurar todos los parámetros necesarios usando la herramienta gratuita **Putty**.

# SSH tradicional vs. EC2 Instance Connect

| | SSH tradicional | EC2 Instance Connect |
|---|---|---|
| Requisito | Requiere abrir el puerto 22 y permitir direcciones IP específicas de forma permanente | Usa una IP temporal de AWS y una clave pública SSH que **expira a los 60 segundos** |
| Seguridad | Menor (la IP autorizada queda abierta de forma indefinida) | Mayor (acceso temporal y controlado) |
| Regla del grupo de seguridad | Origen: IP del usuario (ej. `5.6.7.8/32`) | Origen: rango de IP de AWS (ej. `13.52.6.112/29`), obtenido de https://ip-ranges.amazonaws.com/ip-ranges.json |

# Problemas comunes con SSH en EC2

| Error | Causa | Solución |
|---|---|---|
| 🛑 **"Unprotected private key file"** | El archivo `.pem` tiene permisos demasiado abiertos | Ejecutar en Linux/macOS: `chmod 400 nombre-del-archivo.pem` |
| 🛑 **"Permission denied"** / **"Host key not found"** | Se está usando el usuario incorrecto (ej. `ec2-user` en lugar de `ubuntu`) | Usar el nombre de usuario correcto según el sistema operativo de la AMI |
| 🛑 **"Connection timed out"** | Algo en la red está bloqueando la conexión | Verificar: <br>• El grupo de seguridad permite SSH (puerto 22) desde tu IP <br>• El NACL no está bloqueando el tráfico <br>• La instancia tiene IP pública <br>• Existe una ruta a Internet en la tabla de rutas (a través del IGW) <br>• La instancia no está sobrecargada (CPU al 100%) |

# Escalado vertical (cambiar el tipo de EC2)

Consiste en desplegar la aplicación o base de datos en una **instancia mayor**:

- Disco duro más grande.
- CPU más rápida.
- Más RAM, CPU, E/S o capacidades de red.

En AWS se puede **aumentar el tamaño de una instancia EC2** fácilmente, aunque existen **límites** al escalado vertical.

**Ejemplo de rango de escalado:**

```
De: t2.nano       → 0,5 GB RAM, 1 vCPU
A:  u-12tb1.metal → 12,3 TB RAM, 448 vCPUs
```

# Opciones de compra de instancias EC2

## 1. Instancias bajo demanda (On-Demand)

- Pagas solo por lo que usas, **sin compromiso a largo plazo**.
- Se puede aumentar o disminuir el uso según necesidad.
- Disponibles inmediatamente, se pueden lanzar en cualquier momento.
- Recomendadas para **cargas de trabajo a corto plazo e impredecibles**.

## 2. Planes de ahorro (Savings Plans)

- Descuento por uso a largo plazo (hasta **72%**) a cambio de comprometerse a gastar X USD/hora durante **1 o 3 años**.
- Las compras **no se pueden cancelar ni reembolsar**.
- Opciones de pago: adelantado total, parcial o sin pago adelantado.
- Flexibilidad para cambiar entre instancias dentro de una misma familia (ej. m5.xlarge ↔ m5.2xlarge) y de sistema operativo (Linux/Windows).
- Ideal para cargas de trabajo **predecibles y estables**.

## 3. Instancias reservadas

- Hasta **72% de descuento** frente a pago bajo demanda.
- Se reservan atributos específicos: tipo de instancia, región, ocupación, sistema operativo.
- Periodo de reserva: **1 año** (+descuento) o **3 años** (+++descuento).
- Opciones de pago: sin pago inicial (+), pago inicial parcial (++), pago inicial total (+++).
- Recomendadas para aplicaciones de **uso constante** (ej. una base de datos).

### Estándar vs. Convertible

| Característica | Estándar | Convertible |
|---|---|---|
| Descuento medio (1 año / 3 años) | 40% / 60% | 31% / 54% |
| Cambiar AZ, tamaño de instancia (SO Linux), tipo de red | ✅ (consola / API `ModifyReservedInstances`) | ✅ (consola / API `ExchangeReservedInstances`) |
| Cambiar familia de instancias, SO, tenencia y opción de pago | ❌ | ✅ |
| Beneficiarse de reducciones de precio | ❌ | ✅ |

> Precios actualizados: https://aws.amazon.com/es/ec2/pricing/reserved-instances/

## 4. Instancias Spot

- Descuento de hasta **90%** frente a demanda: las instancias **más económicas** de AWS.
- Pueden **perderse en cualquier momento** si el precio spot supera el precio máximo definido.
- Útiles para cargas de trabajo **tolerantes a fallos**:
  - Trabajos por lotes (Batch Jobs)
  - Análisis de datos
  - Cargas de trabajo distribuidas
  - Tareas con hora de inicio y fin flexible
- **No recomendadas** para trabajos críticos o bases de datos.

## 5. Hosts dedicados (Dedicated Hosts)

- Servidores físicos completos, **dedicados a una organización**, capaces de alojar una o varias instancias EC2.
- La capacidad **no se comparte** con otros clientes.
- Opciones de compra:
  - **Pago por uso**: por segundo de capacidad, sin compromiso.
  - **Reservas**: contrato de 1 o 3 años.
- Es la **opción más cara**.
- Útil para empresas con necesidades fuertes de **regulación o cumplimiento normativo**.

## 6. Instancias dedicadas (Dedicated Instances)

- Se ejecutan en hardware dedicado a un único cliente.
- Pueden compartir hardware con **otras instancias dedicadas de la misma cuenta** (pero no con otras cuentas).
- Ideal para cumplir requisitos de cumplimiento que no permiten multi-tenencia.

### Host dedicado vs. Instancia dedicada

| Característica | Host dedicado | Instancia dedicada |
|---|---|---|
| Facturación | Por host | Por instancia |
| Visibilidad de sockets, núcleos e ID de host | ✅ Sí | ❌ No |
| Afinidad de instancia y host (mismo servidor físico en el tiempo) | ✅ Sí | ❌ No |
| Colocación de instancia dirigida | ✅ Sí | ❌ No |
| Recuperación automática de instancia | ✅ Sí | ✅ Sí |
| Bring-Your-Own-License (BYOL) | ✅ Compatible | ⚠️ Compatibilidad parcial |
| Reservas de capacidad | ❌ No | ✅ Sí |

## 7. Reservas de capacidad a petición (On-Demand Capacity Reservations)

- Reserva capacidad de instancias bajo demanda en una **región o AZ específica**, durante cualquier periodo de tiempo.
- Se puede crear y eliminar **en cualquier momento**, sin compromiso de permanencia.
- Se puede combinar automáticamente con instancias reservadas para aprovechar descuentos por compromiso.
- Adecuada para cargas de trabajo **ininterrumpidas a corto plazo** que necesitan estar en una AZ específica.

# Resumen: opciones de compra de instancias EC2

| Tipo de instancia | Finalidad |
|---|---|
| **Instancias bajo demanda** | Cargas de trabajo de corta duración, precios predecibles, facturación por segundo |
| **Planes de ahorro** | Compromiso de volumen de uso a largo plazo; ideal para cargas continuas y predecibles |
| **Instancias reservadas** | Descuento significativo a cambio de compromiso de 1 o 3 años; cargas continuas |
| **Instancias Spot** | Cargas esporádicas y flexibles a bajo coste; pueden interrumpirse |
| **Hosts dedicados** | Servidores físicos dedicados; control sobre ubicación y hardware |
| **Instancias dedicadas** | Aislamiento a nivel de hardware; ningún otro cliente comparte el hardware |
| **Reservas de capacidad a petición** | Reservar capacidad de cómputo en una región/AZ durante cualquier periodo |

## Analogía del hotel 🏨

| Opción de compra | Analogía |
|---|---|

| **Reservas de capacidad a petición** | Reservar y pagar por una habitación específica para asegurarte de que estará disponible cuando la necesites, aunque no la ocupes todo el tiempo |

# Conceptos clave para recordar

✅ EC2 = Elastic Compute Cloud, IaaS de AWS para alquilar capacidad de cómputo.

✅ El **User Data** ejecuta un script solo en el primer arranque, como usuario root.

✅ El nombre de una instancia (ej. `t3.2xlarge`) indica clase, generación y tamaño.

✅ Los **grupos de seguridad** son firewalls virtuales: solo permiten, nunca deniegan explícitamente.

✅ Por defecto: entrada bloqueada, salida permitida.

✅ **Timeout** → problema de grupo de seguridad. **Conexión rechazada** → problema de la aplicación.

✅ **EC2 Instance Connect** usa claves temporales (60 segundos) y es más seguro que el SSH tradicional.

✅ Desde el 1 de febrero de 2024, AWS cobra por cada IPv4 pública (~$3.6/mes).

✅ Las **instancias Spot** son las más baratas pero pueden interrumpirse en cualquier momento.

✅ Las **instancias reservadas convertibles** permiten cambiar familia de instancia, SO y aprovechar bajadas de precio; las **estándar**, no.

# Preguntas típicas de examen

### ¿Qué significa EC2?

**Elastic Compute Cloud.**

---

### ¿Con qué usuario se ejecuta el script de User Data?

Con el usuario **root**, y solo se ejecuta **una vez**, en el primer arranque de la instancia.

---

### ¿Qué tipo de reglas contienen los grupos de seguridad?

Solo reglas de **permiso** (allow), nunca de denegación explícita.

---

### Un usuario no puede acceder a la aplicación y recibe un timeout, ¿cuál es la causa más probable?

Un problema de configuración del **grupo de seguridad** (puerto no abierto o IP no permitida).

---

### ¿Qué diferencia principal hay entre SSH tradicional y EC2 Instance Connect?

EC2 Instance Connect usa una **clave pública temporal** que expira a los 60 segundos y un rango de IP de AWS, lo que lo hace más seguro que mantener el puerto 22 abierto permanentemente a una IP fija.

---

### ¿Qué tipo de instancia elegirías para una carga de trabajo tolerante a fallos y con el menor coste posible?

**Instancias Spot.**

---

### ¿Qué tipo de instancia elegirías para una base de datos de uso constante durante 3 años?

**Instancias reservadas** (o un **plan de ahorro**, según la flexibilidad necesaria).

---

### ¿Cuál es la diferencia entre un host dedicado y una instancia dedicada?

El **host dedicado** factura por servidor físico completo y ofrece visibilidad y control total sobre sockets, núcleos y colocación de instancias. La **instancia dedicada** factura por instancia, se ejecuta en hardware dedicado a la cuenta, pero sin esa visibilidad ni control adicional.

---
